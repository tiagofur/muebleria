**Análisis + Plan detallado para el Issue #101 (y su contexto con #102, #103, #105, #107)**

Entendí perfectamente tu nota. Tienes razón: **no debemos apresurarnos a poner "Componentes" en cualquier lado**. Hay que hacerlo bien desde el principio porque este es uno de los pilares más importantes para todo lo que viene después (paramétricos en Fase 4, previews 2D/3D, perforaciones CNC, nesting enriquecido, etc.).

Voy a hacer un análisis honesto del estado actual y luego te propongo el diseño que creo más idóneo, futuro-proof y alineado con lo que ya existe en el repo (issues #101 al #112).

### 1. Estado actual de la app (resumen relevante)

Actualmente el modelo es **plano**:

- `Module` tiene:
  - `boardParts: BoardPart[]` (cada uno es un rectángulo con `lengthMm`, `widthMm`, `quantity`, `edges` (4 flags), `optionRole`, descripción)
  - `hardwareLines: HardwareLine[]`
- Todo es **fijo** (no paramétrico).
- No existe el concepto de **Componente reutilizable**.
- No existe **Estructura**.
- No existe clasificación de **posición/placement** (Base, Lateral, Frontal, etc.).
- No hay geometría rica ni metadatos de perforaciones.
- El `optionRole` es la única forma actual de asignar materiales/herrajes.

Esto funciona bien para el MVP actual, pero **es un cuello de botella** para todo lo que quieres hacer después. Cada vez que creas un gabinete de 1 puerta, 2 puertas, 3 puertas, etc., estás repitiendo los laterales, la base, la trasera, etc.

### 2. Tu propuesta para #101 — Mi evaluación

Tienes toda la razón en estos puntos clave:

- **Solo permitir Componentes en Muebles y Estructuras** → Correcto. No queremos que un Componente viva suelto.
- **Primero crear el Componente** (ej: "Lateral Izquierdo") y luego añadirlo a la estructura/mueble → Sí. Este orden es importante.
- **El Componente debe poder llevar geometría** (para 3D futuro) + **perforaciones** (para CNC futuro).
- **El tema de Posición (Base / Lateral / Frontal)** es crítico. Sin esto, el 3D y hasta los previews 2D van a ser muy difíciles de armar correctamente.
- **El tema de Posición2 (Ejes X, Y, Z)** es crítico. Sin esto, el 3D y hasta los previews 2D van a ser muy difíciles de armar correctamente.

Issue #101 ("Componentes reutilizables (puerta, entrepaño…)") + #102 ("Mueble compuesto: estructura + componentes + medidas") + #103 (UI de Ingeniería) forman un **cluster muy importante**. No son features aisladas.

### 3. Diseño que propongo (el más futuro-proof)

#### A. Nueva entidad principal: `Component`

```ts
export type ComponentPlacement =
  | 'base'              // inferior / suelo
  | 'superior'          // tapa de arriba
  | 'lateral_izquierdo'
  | 'lateral_derecho'
  | 'frontal'
  | 'trasera'
  | 'interno'           // entrepaños, divisores verticales
  | 'puerta'
  | 'frente_cajon'
  | 'custom';           // para casos raros

export interface ComponentGeometry {
  readonly type: 'rectangular_board'; // extensible después
  readonly lengthMm: number;
  readonly widthMm: number;
  readonly thicknessMm: number;
  // Futuro:
  // outline?: PolygonPoints;
  // defaultHoles?: Perforation[];
}

export interface Perforation {
  readonly id: string;
  readonly relativePosition: { x: number; y: number; z?: number };
  readonly diameterMm: number;
  readonly depthMm: number;
  readonly type: 'through' | 'blind' | 'dowel' | 'shelf_pin' | 'hinge_cup' | ...;
}

export interface Component {
  readonly id: string;
  readonly code: string;
  readonly name: string;
  readonly placement: ComponentPlacement;        // placement por defecto
  readonly geometry: ComponentGeometry;
  readonly defaultEdges?: readonly EdgeAssignment[];
  readonly perforations?: readonly Perforation[]; // puede estar vacío al inicio
  readonly optionRoles: readonly string[];       // qué grupos de opciones aplican
  readonly notes?: string;
}
```

#### B. Cómo se usa dentro de Mueble / Estructura

En lugar de (o además de) `boardParts` planos, introducimos:

```ts
export interface ModuleComponentInstance {
  readonly componentId: string;
  readonly quantity: number;
  readonly placementOverride?: ComponentPlacement;
  readonly overrides?: {
    dimensions?: Partial<ComponentGeometry>;
    // transform para 3D futuro
    position?: { x: number; y: number; z: number };
    rotation?: { x: number; y: number; z: number };
  };
}

export interface Module {
  // ... campos actuales
  readonly components?: readonly ModuleComponentInstance[]; // nuevo
  readonly structureId?: string; // referencia a una Estructura (issue #102)
}
```

**Estructura** (de issue #102) podría ser algo muy similar a `Module`, pero con semántica de "carcasa / esqueleto". O podemos hacer que `Module` tenga un campo `type: 'simple' | 'compuesto'` y una `estructura` embebida. Yo prefiero entidad separada `Estructura` porque da más claridad semántica.

#### C. Resolución (BOM)

Creamos una función pura en domain:

```ts
resolveModuleToBoardParts(
  module: Module,
  optionChoices: OptionChoices,
  componentsCatalog: Component[]
): ResolvedBoardPart[]
```

Esto expande las instancias de componentes + aplica overrides + optionRoles.

### 5. Ideas adicionales importantes

- **Herencia / Composición**: Un Componente puede referenciar a otro (ej: una "Puerta" puede usar un "Marco" + "Panel"). Esto prepara paramétricos.
- **Posición a dos niveles**:
  - En la definición del `Component` → `defaultPlacement`
  - En la instancia (`ModuleComponentInstance`) → `placementOverride` + transform (para casos especiales o 3D)
- **Perforaciones**: Empezar como metadata opcional. Después el resolver puede generar datos para CNC o nesting.
- **Versionado** (issue #108): Las Estructuras y Componentes deberían poder versionarse porque un cambio en un "Lateral Izquierdo estándar" puede afectar muchos muebles.
- **Parámetros a nivel proyecto** (issue #109): Muy útil después (ej: "todos los laterales de esta cocina usan 18mm").
